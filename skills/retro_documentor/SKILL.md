# Universal Retro Documentor Engine: Enterprise Client-Facing Standard

The **Universal Retro Documentor Engine** is an enterprise-grade, stack-agnostic hyper-intelligence system designed to autonomically reverse-engineer, synthesize, and document any codebase into a high-fidelity, client-ready "System Archive". It bridges the gap between deep technical implementation and high-level Executive Business Value.

---

## 🛠️ The Absolute Enterprise Protocol

Every documentation artifact generated by this engine—whether representing a **Project Root**, a **Module Directory**, or an individual **Source File**—MUST address all sections of the Enterprise Systemic Standard. **No subsets, shortcuts, or omissions are permitted.**

### 1. Enterprise Tone & Language Mandate
- **Tone**: Strictly authoritative, objective, and professional (Consulting Grade / e.g., McKinsey, Gartner, Big 4).
- **Format**: Zero markdown syntax errors. Tables must be perfectly aligned. No conversational AI filler (e.g., "Here is the documentation...").
- **Business Alignment**: Every technical component must explicitly state its return on investment (ROI) or operational business impact.

### 2. The "Zero-Touch / No-Hallucination" Guardrail
- **No Placeholders**: Never output `[Placeholder]`, `TBD`, or `N/A`. 
- **Inference Verification**: If specific metrics (like exact hardware usage) are unavailable, DO NOT GUESS. Use professional deflection: *"Metrics deferred to dynamic observability suite"* or *"Requires SME verification against production telemetry."*
- **Absolute Confidence**: State facts definitively based on source code presence (`pom.xml`, DDL, application configs). 

---

## 🦾 Autonomous Discovery Protocols (ADP)

Every documentation lifecycle follows these stack-agnostic discovery steps to find "Ground-Truth" before rendering:

### 1. Executive Orchestration (Sections 1-7)
- **Web/Microservices**: Identify Controllers, API Definitions, Routes.
- **Batch/Legacy**: Identify JCL, CronJobs, Mainframe Entry Points.
- **Inversion**: Map technical code paths to Client-Facing Business Value.

### 2. Deep Infrastructure & Compliance (Sections 8-14)
- **Data Persistence**: Scan DDL (SQL), NoSQL schemas, ORM configurations.
- **Compliance Scans**: Trace security perimeters, RBAC, Data Localization/Residency vectors, and Disaster Recovery readiness.

### 3. Verification & Modernization Strategy (Sections 15-23)
- **Modernization Rubric**: Standardized evaluation of technical debt, component coupling, and cloud-native migration readiness.
- **Architectural Decision Records (ADR)**: Synthesize *why* an architecture was chosen, effectively justifying the codebase to stakeholders.

## 📂 Shadow Mirror Policy

- **Shadow Mirror Root**: All documentation strictly resides in a `documentation/` shadow-root.
- **Symmetric Architecture**: Root ➜ 1-23; Folder README ➜ 1-23; Source File .md ➜ 1-23.

## 📊 Visual Engineering Standards (Mermaid)

- **Strict Validation Gate**: All generated Mermaid diagrams must pass internal mental-validation before output.
- **Safety Syntax Rules**: 
  1. ALL nodes must have explicit IDs and Labels: `NodeID["Clean Label String"]`.
  2. ALL relationship edges must have quoted labels: `NodeA -- "Specific Action" --> NodeB`.
  3. No HTML tags or unescaped characters in labels.
  4. Place all `style` and `classDef` rules absolutely at the END of the diagram block.
- **Categories**: 
  - *Component Topology*: Global or module-level structural relationships.
  - *Functional Execution Trace*: Detailed, method-to-method sequenced logic paths.
